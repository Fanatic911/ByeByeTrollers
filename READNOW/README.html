1. Commands

All commands use the configured prefix (default +), not a slash. If your prefix is +, commands look like +docs, not /docs.

Default wordPermissionDescription
+sendpanelAdmin or ModeratorSends the verification panel with the Verify button in the current channel.
+verify @user <reason>Admin or ModeratorManually verifies a user, assigns the verified role, and logs the action.
+getinfo @userAdmin or ModeratorShows the IP address currently linked to that Discord account, plus its recent verification attempts.
+getinfo <id>Admin or ModeratorSame as above, using a raw Discord user ID instead of a mention.
+getinfo <ip>Admin or ModeratorShows which account currently owns that IP, and every account that has ever attempted verification from it, with Accepted/Refused status and the reason for each attempt.
+docsAdmin or ModeratorShows this list of commands directly in Discord.
+settingsAdminSends a private, single-use link to the web settings panel via DM.

Renaming a command word

You can change the word typed after the prefix for each command without touching any code, using the commands section of settings.json, or directly from the Commands section of the web settings panel:

"commands": {
  "sendpanel": "panel",
  "verify": "verify",
  "docs": "help",
  "settings": "config",
  "getinfo": "lookup"
}

With the example above and prefix +, moderators would type +panel, +help, and +lookup. The keys on the left must stay exactly as shown (sendpanel, verify, docs, settings, getinfo) — only the values on the right can be changed.

getinfo output example

+getinfo @Someone
> Discord Account: 123456789012345678
> Currently Linked IP: 203.0.113.42
> Recent Attempts:
>   203.0.113.42 — ✅ Accepted — Verified successfully
>   198.51.100.7 — ❌ Refused — Alternate account detected

+getinfo 203.0.113.42
> IP Address: 203.0.113.42
> Currently Linked Account: 123456789012345678
> Attempt History:
>   @Someone (123456789012345678) — ✅ Accepted — Verified successfully
>   @AltAccount (998877665544332211) — ❌ Refused — IP already linked to account: 123456789012345678
If a user clicks the Verify button in the panel and already has the Verified role, the bot replies "You are already verified" instead of generating a new link.
2. settings.json — General

Every option below can be changed either by editing settings.json directly, or from the web panel opened with +settings. The file is validated on every startup and every save — invalid values are rejected with a clear error message instead of silently breaking the bot.

KeyTypeDescription
prefixstringThe command prefix, e.g. +.
commandsobjectRenames the word typed after the prefix for each command. See section 1.
localbooleanIf true, every generated link (verification links, settings panel links) always points to http://localhost:<port> and the public domain reachability check is skipped entirely — no network call is made to test it. Use this while developing locally. Set to false to use your real domain.
public_domainstringYour public domain (e.g. verify.example.com). Only read when local is false. If it cannot be reached, links fall back automatically to http://localhost:<port>.
verification_enabledbooleanTurns the whole verification flow on or off.
EnablebooleanMaster switch. If false, all roles (verified and unverified) are stripped from every member and the system stops responding.
logs_enabledbooleanTurns logging to Logs_Verify_Webbhook on or off.
NotVerified_RolestringRole ID given to members before they verify.
Verified_RolestringRole ID given to members after they verify.
admin_rolearrayRole IDs with full administrative access, including +settings.
moderator_rolearrayRole IDs with access to +verify, +docs, +getinfo, and +sendpanel.
verification_code_expirynumberMilliseconds a verification link stays valid. Default 1800000 (30 minutes).
Discord_Server_LinkstringInvite link shown on the verification page and used as a fallback destination.
Discord_Channel_LinkstringChannel link used for the post-verification redirect.
Privacy_Policy_LinkstringURL to your Privacy Policy. Shown as a link on the verification and landing pages, and in the panel embed. Leave empty to hide the link everywhere.
Terms_Of_Service_LinkstringURL to your Terms of Service. Shown alongside the Privacy Policy link. Leave empty to hide the link everywhere.
Success_Verify_WebhookstringDiscord webhook URL called on a successful verification.
Error_Verify_WebhookstringDiscord webhook URL called when a verification attempt is blocked or fails.
Logs_Verify_WebbhookstringDiscord webhook URL used for general command and moderation logs.
Never put the Discord bot token or captcha secret keys in this file. Those belong only in key.env, which is not committed to version control.

blocked_countriesarrayISO 3166-1 alpha-2 country codes blocked from verifying. This replaces the old separate CountryBlocked.json file — everything now lives in settings.json.

6. sanction.json

Chooses the action applied automatically when a check fails. Every value must be exactly "Kicked" or "Banned".

KeyTriggered when
Alt_DetectThe account looks like an alternate account of an already-verified user.
Ip_Already_LinkedThe IP address is already linked to a different Discord account.
Vpn_Proxy_DetectedThe connection is identified as a VPN, proxy, or hosting IP.
Bot_DetectedThe Discord account itself is a bot account.

Note that an unrecognized browser (section 10) does not trigger a sanction by default — it only blocks that attempt.

7. Database files
FileContent
Database/IP.jsonCurrent mapping of userId → IP address — the single active IP tied to each verified account.
Database/Attempts.jsonFull attempt history, keyed by IP address. Every verification attempt — accepted or refused — is appended here with the account that tried it, the result, the reason, and the timestamp. This is what powers +getinfo.
{
  "203.0.113.42": [
    { "userId": "123456789012345678", "result": "Accepted", "reason": "Verified successfully", "timestamp": "2026-07-04T18:02:11.000Z" },
    { "userId": "998877665544332211", "result": "Refused", "reason": "IP already linked to account: 123456789012345678", "timestamp": "2026-07-04T18:05:44.000Z" }
  ]
}

Each IP keeps at most its 200 most recent entries to keep the file bounded in size.

8. Captcha system

Four captcha modes are supported, selected with capchat_type:

TypeExternal keys neededNotes
localNoneGenerates a random math question (addition or subtraction, values 1–20) and renders it with each character individually rotated, scaled, and offset using CSS transforms — a distorted look without needing any image or canvas library.
recaptchaRECAPTCHA_PUBLIC / RECAPTCHA_PRIVATEGoogle reCAPTCHA v2 checkbox widget.
hcaptchaHCAPTCHA_SITE / HCAPTCHA_SECREThCaptcha checkbox widget.
turnstileTURNSTILE_SITE / TURNSTILE_SECRETCloudflare Turnstile widget.

If a third-party provider's secret key is missing or still set to its placeholder value, the bot fails cleanly with an internal error page and logs the exact reason to the logs webhook — it never sends a broken request to the provider.

9. Anti-VPN / Anti-Proxy

Every verification attempt has its IP checked against a reputation database, which reports both proxy status and hosting/datacenter status — a connection is flagged if either is true, since most abuse comes from either category.

  • Configurable timeout — Vpn_Check_Timeout_Ms controls how long the lookup is allowed to take before giving up.
  • Configurable fail behavior — Vpn_Check_Fail_Closed decides what happens if the lookup itself fails (network error, provider down, timeout). By default the bot fails open (lets the user through) to avoid blocking everyone during a provider outage; set it to true to fail closed instead (treat lookup failures as a detected proxy), which is stricter and recommended for high-risk servers.
  • Sanctions — a detected VPN/proxy triggers the Vpn_Proxy_Detected sanction from sanction.json in addition to blocking the attempt.
  • Full audit trail — every VPN/proxy refusal is recorded in Database/Attempts.json with the reason, so you can review patterns with +getinfo.
No free IP-reputation source is 100% accurate. Treat this as a strong deterrent, not a perfect filter — combine it with Allow_Ip_Already_Linked, Allow_Alt, and the sanction system for defense in depth.
10. Browser blocking (BlockingUnknownNavigator)

When BlockingUnknownNavigator is set to true, the bot reads the browser's User-Agent header and only allows the verification to proceed if the detected browser is present in Authorised_Navigators. Anything unrecognized — headless browsers, unusual automation tools, custom HTTP clients pretending to be a browser without a matching signature — is blocked before the captcha step is even reached.

Recognized browser names you can put in Authorised_Navigators:

NameNotes
ChromeGoogle Chrome and most Chromium-based browsers without their own distinct signature.
FirefoxMozilla Firefox.
SafariApple Safari on macOS and iOS.
EdgeMicrosoft Edge (Chromium-based).
OperaOpera desktop and mobile.
BraveBrave browser.
VivaldiVivaldi browser.
Samsung InternetDefault browser on Samsung Android devices.
UC BrowserUC Browser, common on mobile in some regions.
DiscordDiscord's own in-app browser, used when a user opens the verification link without leaving the Discord mobile app.
If your members mostly open the verification link from inside Discord's mobile app, make sure Discord stays in the list, or they will be blocked.
A refused attempt due to browser blocking is recorded in Database/Attempts.json with the detected browser name as the reason, and the user receives a DM explaining what happened.
11. Rate limiting

Two independent layers of anti-spam protection:

LayerSettingBehavior
Web requestsrate_limit.window_ms / rate_limit.max_requestsEach IP address may make at most max_requests requests to the verification page (GET or POST) within any window_ms window. Once exceeded, further requests receive a "Too Many Attempts" page with HTTP status 429 until the window resets.
Discord buttonrate_limit.button_cooldown_msEach Discord user must wait this many milliseconds between clicks of the Verify button before a new link is generated, preventing DM spam from repeated clicking.

Defaults: a 60 second window, 10 requests per window, and a 15 second button cooldown. All three are editable in the Rate Limiting section of the web settings panel.

14. Security hardening notes

Included by default

  • Secrets isolation: tokens and keys live only in key.env, which is gitignored.
  • Role-based command access, enforced by admin_role and moderator_role.
  • Verification links expire after verification_code_expiry milliseconds.
  • Each verified IP is tied to a single Discord account, with configurable handling of VPNs, proxies, alternate accounts, blocked countries, and now unrecognized browsers.
  • Full audit trail of every accepted and refused attempt, per IP and per account, in Database/Attempts.json, queryable with +getinfo.
  • Two layers of rate limiting to prevent spam and abuse of the verification endpoint and the Discord button.
  • Outbound calls (captcha verification, IP-reputation lookups) use bounded, configurable timeouts.
  • Strict validation of settings.json on every startup and every save, with defaults automatically applied for anything omitted.
  • Discord interactions are deferred immediately and wrapped in error handling; the process also installs unhandledRejection and uncaughtException handlers.
  • Verification and settings-panel links use your configured domain when reachable and fall back to http://localhost:<port> automatically otherwise, or immediately when local is true.

You still need to do this yourself

  • Rotate any secret that has ever been pasted into a chat, ticket, or shared document.
  • Confirm key.env is never committed (git status should never show it).
  • Serve the site over HTTPS in production (see Installation/Debian.sh for an Nginx + Certbot setup).
  • Restrict your firewall to ports 22/80/443.
  • Run npm audit periodically.
  • Back up the Database/ folder regularly — it is the only record of who verified from where.
  • Grant the bot only the Discord permissions it actually needs — never Administrator.

Known limitations

  • Free IP-reputation lookups are rate-limited and not guaranteed to be fully accurate.
  • Browser detection is based on the User-Agent header, which a determined attacker can forge; treat BlockingUnknownNavigator as one layer among several, not a silver bullet.
  • Rate limiting is in-memory and per-process; it resets if the bot restarts and does not share state across multiple server instances.
15. Step-by-step setup

Requirements

  • Node.js v18 or higher, npm v9 or higher
  • A Discord bot token
  • A domain pointed at your server (optional if you plan to use local: true)
  • A reverse proxy such as Nginx or Caddy for HTTPS (recommended for production)

Step 1 — Install Node.js

curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs

On Windows, download the installer from nodejs.org. Verify with node -v and npm -v.

Step 2 — Install dependencies

npm install

Step 3 — Configure key.env

Copy key.env.example to key.env and fill in your bot token and, if needed, your captcha provider keys. See section 5.

Step 4 — Configure settings.json

Set your role IDs, captcha type, domain (or local: true for local testing), security policies, and webhooks. See sections 2 to 4. The file is validated automatically — the bot refuses to start and prints the exact problem if something is missing or the wrong type.

Step 5 — Configure sanction.json

See section 6.

Step 6 — Discord bot permissions

In the Developer Portal, enable the Server Members Intent and Message Content Intent. In your server, the bot needs Manage Roles, Kick Members, Ban Members, Send Messages, and Read Messages.

Step 7 — Start the bot

npm start

The web server starts on the configured port (default 3001); the bot connects to Discord automatically.

Step 8 — Reverse proxy (recommended for production)

server {
    listen 80;
    server_name your.domain.here;

    location / {
        proxy_pass http://127.0.0.1:3001;
        proxy_set_header Host $host;
        proxy_set_header X-Forwarded-For $remote_addr;
    }
}
sudo apt install certbot python3-certbot-nginx
sudo certbot --nginx -d your.domain.here

Step 9 — Send the verification panel

+sendpanel

This posts the verification embed with the Verify button in the current channel.

16. Troubleshooting

"The keys are invalid" error on the verification page

This means capchat_type is set to a third-party provider (recaptcha, hcaptcha, or turnstile) but the matching secret in key.env is missing or still a placeholder. Either fill in the real keys, or switch capchat_type to local, which needs no keys at all.

Links point to the wrong address

Check local in settings.json. If it is true, every link always uses http://localhost:<port> regardless of public_domain. Set it to false and fill in public_domain to use your real domain.

A legitimate user got blocked as an unknown browser

Add their browser name to Authorised_Navigators, or set BlockingUnknownNavigator to false if this happens often. Discord's mobile in-app browser needs Discord in the list.

Too many "Too Many Attempts" errors from legitimate users

Increase rate_limit.max_requests or rate_limit.window_ms in the settings panel. This usually happens on servers sharing a single outbound IP (school, office, campus Wi-Fi).

How do I see who tried to verify from a given IP?

Run +getinfo <ip>. It lists every account that has ever attempted verification from that address, with an Accepted/Refused status and the exact reason for each attempt.
